PART 1: WHAT IS aTox? — COMPLETE APP BREAKDOWN

1.1 Overview & Identity
aTox is a free, open-source Android messaging application built on top of the Tox protocol. Developed by the evilcorpltd team, it is written primarily in Kotlin (85.8% of codebase) with C (4%) for native toxcore bindings. Its first commit was in May 2019 and it currently stands at v0.8.0 (released November 2024) with 458 stars and 75 forks on GitHub.


1.2 App Architecture (Module Breakdown)
aTox follows a clean modular architecture with three distinct layers:

UI Layer: atox/ — The Android UI module. Contains all Activities, Fragments, ViewModels, and Android-specific resources. This is the presentation layer.
Uses Android Jetpack (ViewModel, LiveData, Navigation)
Handles notifications, camera permissions, audio routing
Implements foreground service for keeping the Tox connection alive
Domain Layer: domain/ — Pure business logic module. Contains use-cases, repository interfaces, and data models. Zero Android dependencies.
Contact management, message handling, file transfer orchestration
Kotlin-only library — no Android framework dependency
Core Layer: core/ — The data + network layer. Wraps tox4j (the JVM bindings to c-toxcore) and handles all Tox protocol communication.
Bridges native C toxcore library to Kotlin via JNI
Manages ToxService (background foreground service)
Handles serialization of Tox profile to disk

1.3 Build System
aTox uses a dual build system: Gradle (standard Android builds) and Bazel (for building the native tox4j C library). The native toxcore must be compiled separately for each target architecture:
aarch64-linux-android (ARM64 — most modern phones)
arm-linux-androideabi (ARMv7 — older 32-bit phones)
i686-linux-android (x86 — emulators)
x86_64-linux-android (x86_64 — newer emulators)

PART 2: THE TOX PROTOCOL — DEEP TECHNICAL ANALYSIS

2.1 What is Tox Protocol?
Tox is a peer-to-peer (P2P) instant messaging and VoIP protocol providing end-to-end encryption with no central servers. Born from the 2013 Snowden revelations and Skype/NSA scandal, it was created to make truly private communication accessible. Every user has a public/private keypair. Tox is strictly the protocol — clients (aTox, qTox, Toxic, etc.) are built on top of it.


2.2 The Tox ID
Each Tox user is identified by a 76-character Tox ID composed of:
32 bytes (64 hex chars) — Long-term Public Key (Ed25519/Curve25519): Your permanent cryptographic identity
4 bytes (8 hex chars) — NoSpam value: Anti-spam token. Can be changed to invalidate old friend requests without changing your identity
2 bytes (4 hex chars) — Checksum: XOR-based error detection


2.3 The Protocol Stack (Bottom-Up)
Tox has a layered architecture. From lowest to highest:

Layer 1: UDP/TCP Transport
Tox prefers UDP (port 33445 by default) for direct P2P connections because it is lower latency and not bottlenecked by TCP congestion. TCP is used as a fallback when UDP is unavailable (e.g., behind strict firewalls). TCP connections are made through TCP relay servers.
UDP: Direct peer-to-peer, lower latency, supports hole-punching
TCP: Relayed through TCP relay nodes, used when direct UDP fails
Transition: Tox automatically upgrades TCP → UDP connections when a direct route becomes available

Layer 2: NaCl Encryption (Crypto Core)
All Tox traffic uses the NaCl (Networking and Cryptography library) / libsodium cryptographic primitives:
Key Exchange: Curve25519 Elliptic-Curve Diffie-Hellman (ECDH)
Encryption: XSalsa20 stream cipher
Authentication: Poly1305 MAC (Message Authentication Code)
Combined: crypto_box (NaCl's authenticated encryption = XSalsa20 + Poly1305)
Perfect Forward Secrecy: Ephemeral session keys are used — past sessions cannot be decrypted even if long-term keys are compromised
Known Vulnerability: In 2017, WireGuard author Jason Donenfeld identified KCI (Key Compromise Impersonation) vulnerability in Tox's handshake due to use of 'homebrew crypto' and raw ECDH values as keys. The TokTok team is redesigning this.

Layer 3: DHT (Distributed Hash Table)
Tox uses a custom Kademlia-inspired DHT (based on BitTorrent's BEP-5 design but heavily modified) for peer discovery without central servers:
Every node has a DHT public key (temporary, changes on restart unless bootstrap node)
Close List: 8 nodes closest to your DHT key (by XOR distance metric), stored per bit-prefix index
Friends List: 8 closest DHT nodes to each friend's key — used to locate friends
Pinging: Nodes are pinged every 60 seconds to verify liveness
Bootstrapping: First connection requires a known bootstrap node with stable DHT key
LAN Discovery: UDP broadcast to 255.255.255.255 and IPv6 multicast FF02::1 on port 33445 every 10 seconds — automatically finds Tox peers on local network

Layer 4: Onion Routing (for ID Lookups)
To prevent metadata leakage (preventing observers from knowing who you are looking for on the network), Tox uses onion routing for Tox ID lookups:
3-hop onion path to query the DHT for a friend's IP — first hop is chosen randomly
Each relay only knows the previous and next hop, not source/destination
Only used for friend discovery, not ongoing message traffic
Temporary key pair generated per bootstrap for DHT connections — prevents tracking across restarts

Layer 5: NAT Traversal / Hole Punching
Since most users are behind NAT routers, Tox implements hole punching to establish direct UDP connections:
Cone NAT: Works automatically through normal DHT ping/response exchange
Restricted Cone NAT: Works via coordinated pings to correct IP — DHT assists
Symmetric NAT: Sequential port guessing — toxcore tries all ports near each returned port (e.g., if peers return 1345, 1347 it tries 1344, 1346, 1348...)
NAT ping requests sent every 3 seconds; holes time out after 6 seconds of no response

Layer 6: TCP Relay Servers
For users who cannot establish direct UDP connections, TCP relay servers act as bridges:
TCP connections between client and relay are encrypted (prevents traffic analysis)
Two friends can initially connect over TCP, then migrate to UDP when hole-punching succeeds
Any Tox client can optionally host a TCP relay (API exposed through tox.h)

Layer 7: net_crypto (Tox Transport Protocol)
Once a route is found (via DHT + hole-punching or TCP relay), net_crypto handles the actual secure session:
Requires knowledge of friend's real public key AND DHT public key before connecting
Uses authenticated encryption: all packets are crypto_box encrypted
MAC ensures data is never corrupted in transit
Handles packet reordering, retransmission for reliable delivery

Layer 8: Messenger (Top Layer)
The highest-level module orchestrates all user-facing features:
Nickname/status message transmission when friends come online
Text message send/receive with lossless delivery guarantee
File transfers: identified by 32-byte file_id (e.g., hash of avatar for deduplication)
Streaming: file_size set to UINT64_MAX indicates unknown/streaming transfer
Group chats: temporary friend_connection entries added for each group member
Friend requests: stored in circular buffer, deduplicated by real public key

2.5 Known Weaknesses of Tox Protocol

Security: IP Exposure: All contacts can see each other's IP addresses — no anonymity
Security: KCI Vulnerability: Key Compromise Impersonation attack is possible (Donenfeld, 2017)
Security: No Formal Audit: Tox cryptography has never been independently audited by cryptography specialists
Stability: Alpha Software: Still considered experimental — not ready for high-stakes use cases
Usability: Offline Messaging: Messages are lost if recipient is offline — no store-and-forward
Usability: No Multi-Device: Single-device only — no way to sync conversations across devices
Usability: No Message Search: No full-text search in most clients
Mobile: No Push Notifications: Requires constant background connection — battery drain on mobile
Features: Group Chat Limitations: Requires all members to be online simultaneously for reliable delivery


PART 3: DESIGNING YOUR OWN CUSTOM PROTOCOL

3.1 Why Build Your Own Protocol?
Instead of Tox, you can design a hybrid protocol that learns from the best open-source protocols while eliminating their weaknesses. The key insight is: you don't need to invent everything from scratch — you can compose proven cryptographic building blocks into a new architecture.


3.2 Reference Protocols to Learn From
Protocol 1: Signal Protocol (Best-in-class E2EE)
The Signal Protocol (by Open Whisper Systems) is the most cryptographically sound messaging protocol in widespread use. WhatsApp, Facebook Messenger Secret Chats, and Google Messages all use it. It has two core components:

X3DH: X3DH (Extended Triple Diffie-Hellman): Initial key agreement. Combines 4 DH operations using identity keys, signed prekeys, and one-time ephemeral prekeys to establish a shared secret. Provides mutual authentication, forward secrecy, and crucially — asynchronous messaging (Bob can receive Alice's message even while offline because prekeys are uploaded to a server in advance)
Double Ratchet: Double Ratchet Algorithm: Ongoing message encryption. Combines a Diffie-Hellman ratchet (new DH exchange with every message round-trip) and a Symmetric-Key ratchet (KDF chain for messages in each direction). Provides: forward secrecy (compromise today can't decrypt yesterday's messages) and break-in recovery / post-compromise security (compromise today heals after new DH exchange)


Protocol 2: Noise Protocol Framework (Flexible Handshakes)
The Noise Protocol Framework (by Trevor Perrin) is a toolkit for constructing secure handshake protocols based on Diffie-Hellman key exchange. It's used in WireGuard VPN, WhatsApp, Tor's onion service handshakes, and more. Benefits:
Formally specified and verified handshake patterns
Pattern library: XX, NK, KK, IK etc. — each with different authentication properties
Composable: combine DH algorithms (X25519, X448) with ciphers (AESGCM, ChaChaPoly) and hashes (SHA256, BLAKE2s)
Post-quantum extensions: Can incorporate ML-KEM (Kyber) or other PQC primitives


Protocol 3: Matrix Protocol (Federation Model)
Matrix (matrix.org) is a federated, decentralized communications protocol. Instead of pure P2P, users register on homeservers that federate with each other (like email). Key learnings:
Federation architecture: users on different servers can communicate — no single point of failure
Room/event model: all state is represented as a directed acyclic graph of events, allowing multi-device sync
Megolm: Matrix's group encryption protocol — more efficient than per-member ratchets for large groups
Olm: Matrix's 1-on-1 encryption (inspired by Signal's Double Ratchet)
Self-hosting: anyone can run a homeserver


Protocol 4: Tox Protocol (P2P Discovery)
From Tox itself, the valuable components to keep:
DHT peer discovery: Kademlia-based distributed lookup without central directory
NAT traversal / hole-punching: essential for mobile P2P
LAN discovery: automatic detection of nearby peers on same WiFi


Protocol 5: XMPP + OMEMO (Proven, Federated)
XMPP is a 25+ year old federated protocol (basis of WhatsApp's early architecture, Google Talk). OMEMO is its E2EE extension (based on Signal Protocol). Key learnings:
XML-based stanzas: well-structured message/presence/IQ model
Extension system (XEPs): hundreds of standardized extensions — a template for your feature set
Carbons: message sync across multiple devices

Protocol 6: libp2p (Modular P2P Networking)
libp2p (from Protocol Labs, used in IPFS and Ethereum 2.0) is a modular networking stack for building P2P applications:
Transport agnostic: TCP, UDP, WebRTC, QUIC, WebSockets all supported
Multiplexing: multiple streams over single connection
Peer identity: cryptographic peer IDs (similar to Tox IDs)
Peer routing: multiple DHT implementations (Kademlia, mDNS, etc.)
Content routing: Publish-Subscribe (pubsub) for group messaging



PART 4: YOUR CUSTOM PROTOCOL DESIGN — "NEXUS PROTOCOL"

4.1 Protocol Overview
We propose naming your custom protocol "Nexus" — a hybrid P2P+optional-federation secure messaging protocol that combines the best elements of Tox (P2P), Signal (cryptography), Matrix (sync), and Noise (handshakes). It addresses every weakness of Tox while being designed mobile-first and globally multilingual.

4.2 Identity System
NexusID — Your Cryptographic Identity
Replace Tox's simple keypair with a richer identity structure:


Your NexusID string = Base58(IK + SPK + SPK_Signature + NoSpam + Checksum) — approximately 140 characters. Human-friendly display using a BIP39-style word list as an alternative ('oak river sunset cloud...' — 12 words = 128-bit identifier).

4.3 Nexus Protocol Stack
Layer 1: Transport (Multi-Protocol)
Primary: UDP with QUIC (modern, multiplexed, built-in TLS 1.3, 0-RTT reconnection, handles IP changes gracefully — ideal for mobile)
Secondary: WebRTC DataChannels (works in browsers, cross-platform P2P with built-in STUN/TURN)
Fallback: TLS 1.3 over TCP (for restrictive networks)
Bridge: Tor hidden services integration (optional, for maximum anonymity)

Layer 2: Nexus Handshake (Noise-Based)
Use the Noise Protocol XX pattern (mutual authentication with identity key hiding from passive observers):
Pattern: Noise_XX_25519_ChaChaPoly_BLAKE2b — meaning X25519 DH, ChaCha20-Poly1305 AEAD, BLAKE2b hash
Extended with: Hybrid PQ using ML-KEM-768 encapsulation — combines classical X25519 with post-quantum KEM
Prologue: Application ID + Protocol version + timestamp hashed in — prevents cross-protocol attacks
Result: Authenticated, forward-secret session channel, resistant to quantum attackers

Layer 3: Key Agreement — Nexus Key Exchange (NKX)
NKX is your X3DH-inspired asynchronous key agreement:
Alice performs 4 DH operations using Bob's IK, SPK, OPK and her ephemeral key
Additionally: Alice encapsulates a secret using Bob's ML-KEM public key (post-quantum)
Combined secret = HKDF(DH1 || DH2 || DH3 || DH4 || KEM_secret) — quantum-resistant master secret
Bob can be offline: his prekeys are distributed via the DHT or optional relay servers
OPK consumed: prevents replay attacks on initial handshake

Layer 4: Message Encryption — Nexus Ratchet (NR)
NR is your Double Ratchet variant with improvements:
DH Ratchet: X25519 DH ratchet (same as Signal) — new DH exchange per message round-trip
Symmetric Ratchet: ChaCha20-Poly1305 with BLAKE2b-based KDF chains
Message Keys: 32-byte per-message symmetric keys, deleted after use (forward secrecy)
Out-of-order: skipped message keys cached for up to 1000 messages
PQ Ratchet: Optional ML-KEM re-encapsulation every N messages for additional PQ security
Header Encryption: Message headers (ratchet public key, message number) are encrypted to prevent metadata leakage to relay servers

Layer 5: Peer Discovery — Nexus DHT (NDHT)
Enhanced Kademlia DHT combining the best of Tox and libp2p:
Node IDs: BLAKE2b(Identity_Public_Key) — deterministic, resistant to Sybil attacks
k-bucket size: 20 nodes (vs Tox's 8) — more robust routing
Encrypted node records: Node IP/port encrypted with contact's public key — IP only revealed to authorized contacts
IP Privacy: Intermediate DHT hops cannot see who you are looking for (onion-style query routing)
LAN discovery: mDNS-based (vs Tox's raw UDP broadcast) — standards-compliant
Bootstrap: Hard-coded + user-configurable bootstrap nodes, plus DNS seed nodes

Layer 6: Offline Delivery — Nexus Store & Forward (NSF)
This is a major improvement over Tox's lack of offline messaging:
Sealed Sender: Messages encrypted so even relay servers cannot determine sender identity
Optional relay network: Volunteer relay nodes store encrypted message envelopes for offline recipients
Envelope format: Double-encrypted — outer layer for relay routing, inner layer is NR-encrypted for recipient only
TTL: Messages expire after configurable time (default 7 days) if not delivered
Multi-device: Each device has its own sub-key (derived from master identity key); messages are encrypted for all registered devices simultaneously

Layer 7: Group Messaging — Nexus Group Protocol (NGP)
Using Megolm-inspired approach for efficient group encryption:
Group Session Key: A symmetric ratcheting session key shared with all members
Distribution: Group session key distributed via pairwise NR-encrypted channels (each member gets a copy)
Sender Key: Each member has a unique sender key — even group members cannot spoof each other's messages
Group size optimization: Unlike per-member Double Ratchet (O(n) messages), Megolm-style is O(1) — send one encrypted message to everyone
Admin controls: cryptographic group management — add/remove members with re-keying

7.2 Cryptographic Export Regulations
Shipping strong cryptography has export control implications. Key points:
US EAR (Export Administration Regulations): Apps using open-source publicly available crypto are generally exempt, but you should file a classification request with BIS (Bureau of Industry and Security) for apps distributed from the US
EU regulations: Generally permissive for publicly available cryptographic software
China/Russia/Iran: Some countries restrict import of strong encryption products — consult local counsel
Safe approach: Use standard, widely-deployed algorithms (AES, ChaCha20, X25519, Ed25519) — these are well-established export classifications
