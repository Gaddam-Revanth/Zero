SECURITY REVIEW NOTES (MUST-FIX SPEC GAPS FOR v1.0)

This document is an overview + partial specification. Several sections are placeholders.
To prevent "spec ambiguity" vulnerabilities, the following requirements are mandatory for
any implementation claiming compliance with ZERO Protocol v1.0:

R1  Transcript Binding:
    The ZKX master secret derivation MUST be bound to the Noise XX transcript hash.
    (Without this, a MITM could potentially splice phases in some implementation variants.)

R2  Downgrade Resistance:
    The Noise XX prologue MUST include: protocol name, version, ciphersuite identifiers,
    and feature flags (PQ on/off, header encryption on/off, etc.). Any mismatch MUST abort.

R3  Replay Protection:
    - ZKX initiation messages MUST include a replay-resistant identifier (e.g., 128-bit random)
      and responders MUST keep a bounded replay cache per sender for a configurable window.
    - ZSF relays MUST reject duplicate envelopes by (recipient_node_id, envelope_id) within TTL.

R4  Key Confirmation:
    After ZKX completes, both sides MUST perform explicit key confirmation (e.g., exchange
    a MAC keyed with a derived confirmation key) before accepting the session as established.

R5  Exact KDF Hash:
    This spec currently mentions BLAKE2b usage in multiple places. Implementations MUST
    specify the HKDF hash function explicitly (e.g., HKDF-SHA512 or HKDF-BLAKE2b-512).
    Mixing implementations with different HKDF hashes will silently break security.

R6  ML-KEM Parameter Consistency:
    The spec mentions ML-KEM-768 (FIPS 203) for handshake, but also references ML-KEM-512
    in PQ ratchet. ML-KEM-512 is not standardized in the same FIPS 203 profile set.
    Either: (a) remove PQ ratchet from v1.0, or (b) standardize on ML-KEM-768 only.

These requirements are incorporated as clarifications below where relevant.


1.1  What is ZERO Protocol?
ZERO Protocol is a peer-to-peer, end-to-end encrypted, post-quantum resistant messaging protocol designed to provide secure, private, and decentralized communication for all people on Earth. Its name reflects its core principles: Zero central servers required, Zero knowledge of your communications by any third party, Zero compromise on privacy or security, and Zero barriers to global access through full support of all world languages.


1.2  Design Goals

1.3  Protocol Name Expansion

1.4  Protocol Versioning
ZERO Protocol uses semantic versioning: MAJOR.MINOR.PATCH. The current specification is ZERO Protocol v1.0.0. Backwards-incompatible changes increment MAJOR. New features increment MINOR. Bug fixes increment PATCH. All clients must advertise their supported protocol version during the ZKX handshake. Clients on incompatible major versions cannot communicate.

1.5  Relationship to aTox and Tox Protocol
ZERO Protocol is inspired by — but not derived from — the Tox Protocol. It shares Tox's core philosophy of serverless P2P communication but replaces every technical component: the cryptographic handshake (fixing Tox's KCI vulnerability), the identity system (adding post-quantum resistance), the peer discovery (adding IP privacy), and adding offline message delivery that Tox entirely lacks. The Android app built on ZERO Protocol is architecturally inspired by aTox's clean modular Kotlin design (atox/domain/core modules), but uses Rust instead of C for the protocol core, and UniFFI instead of hand-written JNI for language bindings.

3.1  Design Principle: No Custom Cryptography

3.2  Complete Cryptographic Primitive Set

3.3  Why Each Algorithm Was Chosen
Ed25519 for Identity Signatures
Ed25519 (EdDSA over Curve25519) was chosen over ECDSA (secp256k1/secp256r1) for three reasons: constant-time implementation is the default (ECDSA requires careful nonce generation to avoid private key leakage), much faster verification (batch verification possible), and smaller signatures (64 bytes vs 71-72 bytes for ECDSA). Ed25519 is used in OpenSSH, TLS 1.3, Signal, and Tor.

X25519 for Diffie-Hellman
X25519 uses the Curve25519 elliptic curve in Montgomery form for ECDH. It was designed by Daniel Bernstein to be fast, safe, and hard to implement incorrectly. Unlike NIST P-256 or P-384, Curve25519 has no suspected NSA manipulation (the curve parameters have a transparent mathematical derivation). Constant-time implementation is straightforward. X25519 is used by TLS 1.3, WireGuard, Signal, and SSH.

ML-KEM-768 for Post-Quantum KEM
ML-KEM-768 was standardized by NIST as FIPS 203 (August 14, 2024) — the primary post-quantum Key Encapsulation Mechanism standard. It is based on Module Learning With Errors (ML-LWE), a structured lattice problem that is hard for both classical and quantum computers. Level 3 (768) provides approximately AES-192 equivalent security. Public key: 1,184 bytes. Ciphertext: 1,088 bytes. This overhead only occurs during the ZKX handshake — ongoing message traffic uses symmetric keys derived from the KEM output.

ChaCha20-Poly1305 for Message Encryption
ChaCha20-Poly1305 was chosen over AES-GCM for message encryption for mobile performance reasons: on devices without AES hardware acceleration (many low-end Android phones), ChaCha20 is 2-3× faster than AES while providing equivalent security. ChaCha20-Poly1305 is the preferred AEAD in TLS 1.3, WireGuard, and QUIC. It has no timing side-channel vulnerabilities.

BLAKE2b for Hashing
BLAKE2b is faster than SHA-256 on 64-bit platforms while providing equivalent security. It was designed as a drop-in replacement for MD5/SHA families with improved security properties. Used in Noise Framework's ck/h state, HKDF salt, and ZERO ID checksums. BLAKE3 (its successor) is even faster but is newer and less widely deployed — BLAKE2b is the conservative choice.


4.1  Overview
ZKX (ZERO Key Exchange) is the asynchronous initial key agreement protocol. It establishes a shared master secret between two ZERO users even when one of them is offline. ZKX combines three cryptographic mechanisms: the Noise XX handshake pattern (for authenticated channel establishment), X3DH (Extended Triple Diffie-Hellman, for asynchronous key agreement using prekeys), and ML-KEM-768 (for post-quantum resistance). The result is a single master secret that is secure against both classical and quantum adversaries.

4.2  Phase 1: Noise XX Handshake
Before ZKX begins, a Noise XX handshake establishes an authenticated, encrypted channel. Noise XX provides mutual authentication (both parties prove they hold the private key corresponding to their published public key) and identity hiding from passive observers.

Noise XX Pattern Execution

ZKX PROLOGUE (REQUIRED)
The Noise XX prologue MUST include, at minimum:
- "ZERO-Protocol" literal
- Protocol version (MAJOR.MINOR.PATCH)
- Ciphersuite identifiers (X25519, ChaCha20-Poly1305, hash, HKDF hash)
- Feature flags (PQ handshake required, header encryption required, etc.)
Any prologue mismatch MUST abort the handshake.

4.3  Phase 2: X3DH Key Agreement (Classical Layer)
After the Noise channel is established, ZKX runs the X3DH component to derive the classical portion of the master secret. X3DH requires Bob to have published his key bundle (SPK + OPKs) to the ZDHT or relay node.

Step 2  Alice verifies SPK_sig using Bob's ISK_pub. If verification fails, ZKX is aborted — potential MITM attack.
Step 3  Alice verifies IDK_pub certification signature using Bob's ISK_pub.

4.5  Master Secret Derivation
The classical and post-quantum secrets are combined via HKDF to produce the ZKX master secret. An attacker must simultaneously break BOTH the classical X3DH layer AND the ML-KEM-768 layer to compromise the key agreement:

MASTER SECRET BINDING (REQUIRED)
Let H_noise be the 32-byte Noise XX handshake transcript hash.
The ZKX master secret derivation MUST include H_noise in the HKDF input key material:

IKM = H_noise || DH1 || DH2 || DH3 || (DH4?) || KEM_shared_secret

confirm_key = HKDF(master_secret, info="ZERO-ZKX-v1-key-confirm", L=32)
Alice sends: MAC(confirm_key, "A->B" || H_noise)
Bob replies: MAC(confirm_key, "B->A" || H_noise)
Any failure MUST abort and delete derived keys.



5.1  Overview

ZR is based on the Double Ratchet Algorithm (Signal, 2016) with additional improvements for privacy against relay servers (header encryption) and optional post-quantum forward secrecy (PQ ratchet step). ZR provides per-message encryption keys, perfect forward secrecy, and post-compromise security.

5.5  Out-of-Order Message Handling
Network packets can arrive out of order. ZR handles this by caching skipped message keys in MKskipped. If message N+5 arrives before N+1 through N+4, ZR derives and caches MK for N+1 through N+4, then uses the MK for N+5 immediately. When N+1 through N+4 arrive later, their cached MKs are retrieved and deleted. Maximum cache: 2,000 skipped keys per session. Keys older than 7 days are automatically purged.

6.1  Overview
ZDHT (ZERO Distributed Hash Table) is the peer discovery mechanism. It enables users to find each other's current IP addresses and network locations without any central directory server. ZDHT is based on Kademlia DHT with significant enhancements for privacy, performance, and Sybil resistance compared to Tox's DHT.

6.3  Privacy-Preserving Node Records
In Tox DHT, every node can see every other node's IP address. In ZDHT, node records containing IP addresses are encrypted such that only authorized contacts can read them:

NODE RECORD REQUIREMENTS (REQUIRED)
- NodeRecord plaintext MUST be signed by the owner's ISK (Ed25519).
- EncryptedNodeRecord MUST use AEAD with AAD bound to NodeID (and protocol version).
- Decrypting clients MUST verify:
  - AEAD authenticity
  - Signature validity
  - Freshness window (e.g., timestamp within +/- 10 minutes, configurable)
  - Monotonic update policy to prevent rollback (store highest timestamp per contact)

6.5  Onion Routing for Friend Discovery
When Alice wants to find Bob's current IP address, she does not query the DHT directly (which would expose to DHT nodes that Alice is looking for Bob). Instead she uses 3-hop onion routing:
Alice selects 3 random ZDHT nodes as onion hops: H1, H2, H3 (H3 closest to Bob's NodeID).
Alice wraps the FIND_RECORD_REQ in 3 layers: outermost encrypted for H1, middle for H2, innermost for H3.
H1 decrypts outer layer, forwards to H2. H1 knows Alice but not Bob.
H2 decrypts middle layer, forwards to H3. H2 knows H1 but not Alice or Bob.
H3 decrypts innermost layer, performs FIND_RECORD for Bob's NodeID, returns result back through the path.
Alice receives Bob's NodeRecord, decrypts the per-contact ciphertext to reveal Bob's current IP:port.

7.1  Primary Transport: QUIC (RFC 9000)
ZERO's primary transport protocol is QUIC v1 (RFC 9000). QUIC was chosen over raw UDP and TCP for its superior mobile characteristics: seamless connection migration (if your IP changes when switching from WiFi to 4G, the QUIC connection continues without interruption), 0-RTT reconnection (reconnects to known peers without a full handshake), built-in TLS 1.3 encryption, and multiplexed streams (no head-of-line blocking).

7.3  Fallback Transport Hierarchy
ZERO tries transport options in order, automatically falling back when the preferred option is unavailable:
QUIC over UDP (preferred) — direct P2P connection via UDP port 44300 (default).
QUIC over UDP (hole-punched) — if direct fails, attempt NAT hole-punching via ZDHT.
WebRTC DataChannel — for browser-based clients; built-in STUN/TURN support.
TCP over TLS 1.3 — for networks that block UDP entirely (some corporate firewalls).
TCP Relay via ZERO relay server — encrypted relay when direct connection impossible.
Tor Hidden Service (onion address) — maximum anonymity mode; user-enabled only.


8.1  The NAT Problem
The vast majority of devices are behind NAT routers that block unsolicited incoming UDP packets. ZERO implements multiple NAT traversal strategies to establish direct P2P connections without relay servers wherever possible. Direct connections have lower latency, no relay bandwidth costs, and higher privacy (relay server cannot observe traffic patterns).

8.2  NAT Types & Success Rates

8.3  ZERO Hole-Punching Algorithm
Step-by-Step Process
Alice and Bob both connect to ZDHT and learn each other's most recently announced IP:port.
Alice sends a coordination packet through ZDHT: 'I will start punching Bob at T+500ms'.
Both Alice and Bob simultaneously send UDP packets to each other's IP:port at T+500ms. This opens a 'hole' in each other's NAT — the router creates an outbound mapping.
Alice and Bob each send 10 UDP probe packets 50ms apart to maximize the chance of the hole being open on both sides simultaneously.
If direct connection established: both parties acknowledge via ZDHT. Begin QUIC session.
For symmetric NAT: additionally try sequential port guessing — if observed external ports are 45301, 45303 try 45298-45310 systematically.
Timeout: if no direct connection after 5 seconds, fall back to TCP relay server.

10.1  Overview — Solving Tox's Biggest Limitation
Tox Protocol's most fundamental limitation is that messages are lost forever if the recipient is offline. ZSF (ZERO Store & Forward) solves this completely. It is ZERO Protocol's offline message delivery system, built on the Sealed Sender concept: relay nodes store encrypted message envelopes for offline recipients, but critically, they cannot determine who sent the message.

10.2  Sealed Sender Design
In ZSF, each message envelope has two layers of encryption:

10.3  ZSF Message Envelope Structure

ZSF ENVELOPE REQUIREMENTS (REQUIRED)
- Two-layer encryption:
  - Outer layer decryptable by the relay (reveals only TTL + opaque inner blob)
  - Inner layer decryptable only by recipient (reveals sender identity + payload)
- Outer ciphertext MUST include a per-envelope nonce and the relay ephemeral public key.
- Inner ciphertext MUST include a per-envelope nonce and the sender ephemeral public key.
- The relay MUST validate PoW before decrypting and before accepting storage.
- Relays MUST apply replay and quota controls per recipient_node_id.

10.5  ZSF Message Delivery Flow
Alice composes a message to offline Bob. ZSF detects Bob is not currently reachable.
Alice locates Bob's preferred ZSF relays from his ZDHT NodeRecord (Bob advertises 3 preferred relays).
Alice solves proof-of-work for anti-spam.
Alice constructs ZSFEnvelope: builds inner sealed layer (only Bob can open), builds outer layer (only relay can open to see recipient NodeID + TTL).
Alice submits envelope to Bob's 3 preferred relays (redundancy). Relays store it.
When Bob comes online, his client queries his preferred relays for pending envelopes.
Bob downloads, decrypts outer layer (confirming the relay chose correct recipient), decrypts inner layer using his IDK_priv, reveals sender and ZR-encrypted message.
Bob sends delivery receipt (signed by his ISK) back through relay to Alice.
Relay deletes the stored envelope on confirmed delivery or TTL expiry.

11.4  Member Onboarding & Removal
Adding a New Member
Admin creates an 'add_member' event signed with GSK: { new_member_ISK_pub, timestamp }.
Admin securely sends new member's inbound sessions to them via pairwise ZR channels (one ZR message per existing member's sender key, all sent to new member).
New member can decrypt messages sent AFTER their join timestamp. They cannot decrypt prior history.
New member's outbound session distributed to all existing members via ZSF-backed pairwise ZR messages.

Removing a Member (Re-key)
Admin creates a 'remove_member' event signed with GSK.
Admin generates new outbound sessions for all remaining members.
New sessions distributed via pairwise ZR to all remaining members.
Removed member's outbound session is invalidated. They can no longer encrypt messages the group will accept.
Removed member still has their inbound sessions for history they already received — they can re-read their local copy but cannot receive new messages.

ZAV (ZERO Audio/Video) provides end-to-end encrypted voice and video calling built on top of ZERO's existing P2P transport. Unlike Tox (which uses custom toxav call signaling), ZAV uses WebRTC media streams for media transport combined with ZERO's own signaling over ZR-encrypted messages.

13.1  Overview
ZFT (ZERO File Transfer) provides reliable, encrypted file transfer with integrity verification, pause/resume support, and streaming capability. All file content is end-to-end encrypted using the ZR session keys — only the intended recipient can decrypt the file.

13.2  ZFT Packet Structure

13.3  File Transfer Flow
Sender: generate transfer_id, compute file_id (BLAKE2b hash of full file). Send ZFTOffer via ZR.
Receiver: display offer with filename, size, thumbnail. User accepts/rejects.
On accept: receiver sends ZFTAccept { transfer_id, start_chunk }. start_chunk > 0 for resume.
Sender: begins sending ZFTChunk messages sequentially over QUIC Stream 1. Flow-controlled by QUIC.
Receiver: verifies each chunk's hash. Stores chunks to disk. Sends ZFTAck every 100 chunks.

After the initial ZKX, reconnections are much faster. Both parties persist their ZR state (root key, chain keys, ratchet state) in the local encrypted database. 15.2  Threat Model
Adversary Types ZERO Protects Against
✅  Passive Network Observer: Sees encrypted traffic only. Cannot determine content, sender, or recipient (with ZSF + sealed sender). Cannot break encryption without quantum computer.
✅  Active Network Attacker (MITM): Cannot substitute keys if ZERO IDs are correctly verified (safety numbers). Noise XX handshake provides authentication.
✅  Malicious Relay Server: Can see timing and approximate message sizes. Cannot read message content. Cannot identify sender (ZSF sealed sender). Cannot impersonate either party.
✅  Malicious DHT Node: Can refuse to provide node records (DoS, not compromise). Cannot forge node records (Ed25519 signatures). Cannot read IP records not intended for them.
✅  Quantum Computer Attacker: ML-KEM-768 (FIPS 203) provides quantum-resistant key agreement. Currently stored traffic cannot be retroactively decrypted.
⚠️ Partial  Compromised Device (after fact): ZR post-compromise security heals after new DH exchange. Attacker with device gets current/future messages until ratchet heals.
✅  Legal Compulsion (server seizure): No user data on central servers. Relay servers only store sealed envelopes they cannot read.
✅ with passphrase  Physical Device Seizure: Local database encrypted with Argon2id-derived key. Optional duress PIN wipes data. Passphrase required.


16.1  Why Post-Quantum Now?
Quantum computers running Shor's algorithm can break Curve25519 (X25519, Ed25519) and all other elliptic curve cryptography in polynomial time. While cryptographically-relevant quantum computers do not yet exist, 'harvest now, decrypt later' attacks are a present threat: nation-state adversaries are recording encrypted network traffic today, storing it, and will decrypt it when quantum computers become available. For a messaging protocol, this means conversations recorded today could be exposed in the future. ZERO Protocol uses NIST-standardized post-quantum algorithms from day one — not as a future upgrade, but as a core requirement.

16.4  The Hybrid Approach — Why Both Classical and PQ
ZERO uses a hybrid scheme combining X25519 (classical) with ML-KEM-768 (post-quantum). The combined secret is: master_secret = HKDF(X25519_secret || ML-KEM_secret). This means: if ML-KEM-768 is somehow broken (e.g., a novel mathematical attack is discovered), the X25519 layer still provides classical security. If X25519 is broken by a quantum computer, the ML-KEM-768 layer still provides quantum security. An attacker must break BOTH simultaneously. This is the approach recommended by NIST and used by Signal's PQXDH (2023).


17.1  Why Rust for ZERO Protocol
ZERO Protocol's core is implemented in Rust for four critical reasons: Memory safety without garbage collection (no buffer overflows, no use-after-free — entire class of C/C++ vulnerabilities eliminated at compile time), zero-cost abstractions (Rust code compiles to machine code as efficient as C), excellent cryptography ecosystem (RustCrypto, libsodium bindings, liboqs), and first-class cross-platform compilation (compile once, run on Android, iOS, Linux, Windows, macOS from same codebase).

20.1  Serialization Format
ZERO Protocol uses CBOR (Concise Binary Object Representation, RFC 8949) as the wire serialization format. CBOR was chosen over Protocol Buffers (Protobuf) for three reasons: it is a published IETF standard (not vendor-controlled), it is self-describing (easier to debug and audit), and it has excellent Rust support via the serde_cbor crate. All ZERO protocol messages are CBOR-encoded before encryption.

We commit to: acknowledging receipt within 48 hours, providing a fix timeline within 14 days, issuing a public CVE and credited disclosure within 90 days of fix release. A bug bounty program will be launched post-v1.0.

C.1  Bootstrap Node Requirements
Bootstrap nodes are the entry points into the ZDHT network. They must be: stable (99.9%+ uptime), well-connected (low-latency connections to multiple internet exchange points), and operated by trusted community members or organizations. Bootstrap nodes do NOT have special privileges — they are ordinary ZDHT nodes that happen to have well-known, stable addresses.
